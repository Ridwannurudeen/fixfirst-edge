Context

The plant floor is the worst environment for a cloud product.

A technician standing at a broken conveyor has four things to search: the equipment manual, the incident log, the parts catalog, and their own memory. None of them are fast over a plant-floor WiFi.

Maintenance copilots that route queries to a vendor cloud fail the second the signal drops. LLM-generated diagnostics are unauditable — a hallucinated fix can't be traced to a page. For pharma, defense, utilities, and GxP-scoped buyers, shipping equipment data to a third-party model isn't a tradeoff, it's a disqualifier.

FixFirst Edge keeps the corpus, the embeddings, the index, and the retrieval on one laptop. Every diagnosis cites the exact manual page, prior incident, and part number it came from.

Inputs

However the problem gets described, one retrieval pipeline handles it.

Error code

Identifier-aware RRF routes exact tokens like E04 through Actian's keyword index; the dense lane catches the symptom phrasing around them.

Photo of a damaged part

CLIP-ViT-B-32 embeds the image on the same machine, then searches the 512-dim image vector lane against every image in the collection. No upload, no analysis API.

faster-whisper transcribes locally. The transcript embeds through the audio vector lane and cross-retrieves text, images, and prior voice notes in a single query.

In the live UI, the technician narrows by machine type or severity. Under the hood, the retrieval path also uses indexed model_no and fault_code metadata — extracted from the query, applied in-engine against Actian's keyword index, not post-filtered in Python.

Stack

Why Actian VectorAI DB is load-bearing here.

Three features in a single database that a commodity vector store would require three tools to stitch together. Each one is doing real work in the request path, not marketing.

01
Named vectors

One collection holds three embedding spaces — text_vec (384d), image_vec (512d), audio_text_vec (384d). A document can carry any subset and be cross-retrieved through any modality. 

vectors_config={
  "text_vec":       VectorParams(size=384),
  "image_vec":      VectorParams(size=512),
  "audio_text_vec": VectorParams(size=384),
}
02
Filtered search

Six keyword-indexed fields — doc_type, machine_type, model_no, fault_code, severity, part_no. Applied in-engine, not post-filtered in Python. 

builder = FilterBuilder()
builder.must(Field("doc_type").eq("manual"))
builder.must(Field("model_no").eq("CX-200"))
03
Hybrid RRF fusion

Dense ANN and an identifier-filtered ANN branch run in parallel, then merge with reciprocal rank fusion. Tokens like E04 win through exact filters; "motor tripped on overload" wins through dense. RRF covers both. 

# reciprocal rank fusion
score(d) = Σ 1 / (60 + rank_i(d))
# i ∈ {dense, id-filtered}

Questions that come up

The six things buyers actually ask.

Can I ingest my own manuals and incident logs? +
Yes. Drop PDFs into data/raw/manuals/, schematic images into data/raw/images/, and CSVs for incidents, parts, and error codes into data/raw/. The bulk-ingest script chunks PDFs with pdfplumber, embeds with bge-small / CLIP / whisper, and upserts to the Actian collection. Re-run it whenever the source documents change.
How large a corpus does this handle on a laptop? +
The benchmarked corpus is small (3 PDF manuals, 30 incidents, 25 parts, 13 error codes, 6 schematics, 5 voice clips) because it's built for the hackathon. Actian VectorAI DB itself scales to hundreds of millions of vectors per collection; the bottleneck on a 16 GB laptop is embedding throughput, not retrieval. A ten-thousand-page manual corpus is well inside the envelope. Beyond that, move embedding to a GPU box and leave retrieval on the laptop.
Does this replace my CMMS? +
No. A CMMS tracks work orders, preventative schedules, spares inventory, and asset hierarchies. FixFirst Edge sits next to the CMMS — it's the retrieval layer a technician opens during the fix. Export your incident log and parts catalog, ingest them here, and the two systems are complementary.
Does it need a GPU? +
No. bge-small, CLIP-ViT-B-32, and whisper tiny.en all run acceptably on CPU — the 843ms p50 number is from a 16 GB laptop, CPU-only, WSL2. A GPU helps on the embed side during bulk ingest but is unnecessary for query-time retrieval.
What's the security posture for regulated environments? +
Everything runs local. The diagnose path has zero outbound network calls — that's verifiable by opening the browser network panel during a query or by running the whole stack in a DMZ with egress blocked. No API keys, no license server, no telemetry endpoints, no update channel. The Docker compose file is the entire supply chain; auditors can read it in an afternoon.
What does it cost? +
Zero. MIT-licensed source, free Actian VectorAI DB for the embedded footprint this needs, and the hardware is whatever laptop the technician already has. For context, a cloud CMMS at $35–$200 per user per month costs a ten-person maintenance team between $4,200 and $24,000 a year.
